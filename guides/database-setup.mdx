The backend uses PostgreSQL 14+ with TypeORM. The schema is managed automatically via `synchronize: true` — no migration files are needed. On first startup, a seed function inserts three test products if the table is empty.

## Prerequisites

- PostgreSQL 14 or later installed and running
- A database user with `CREATE DATABASE` privileges (or an existing database)

---

## Setup steps

<Steps>
  <Step title="Create a database user and database">
    Connect to PostgreSQL as a superuser and run:

    ```sql
    CREATE USER checkout_user WITH PASSWORD 'your_password';
    CREATE DATABASE checkout_db OWNER checkout_user;
    GRANT ALL PRIVILEGES ON DATABASE checkout_db TO checkout_user;
    ```

    Use any username and password you prefer. The values must match the `DB_USER`, `DB_PASS`, and `DB_NAME` variables in `backend/.env`.
  </Step>

  <Step title="Configure backend environment variables">
    Edit `backend/.env` with the connection details:

    ```bash
    DB_HOST=localhost
    DB_PORT=5432
    DB_USER=checkout_user
    DB_PASS=your_password
    DB_NAME=checkout_db
    ```

    See [Environment Variables](/guides/environment-variables) for the full variable reference.
  </Step>

  <Step title="Start the backend">
    TypeORM connects on startup and automatically creates all tables via `synchronize: true`. No `CREATE TABLE` statements or migration commands are needed.

    ```bash
    cd backend
    npm run start:dev
    ```

    The first time the backend starts with an empty database, the seed function runs and inserts three products.
  </Step>
</Steps>

## TypeORM configuration

The TypeORM data source is configured with `synchronize: true`, which means TypeORM reads the entity decorators and automatically creates or alters tables on every startup. This is appropriate for development and this project's scope — there are no migration files to run.

<Note>
For production deployments, consider switching `synchronize` to `false` and generating explicit migrations with `typeorm migration:generate` to maintain full control over schema changes.
</Note>

## Auto-seed on startup

The file `src/database/seeds/product.seed.ts` runs on every startup. If the `products` table is empty (i.e., `COUNT(*) === 0`), it inserts three products:

| Product | Price | Stock |
|---------|-------|-------|
| Audífonos Bluetooth Pro | $150.000 COP (15 000 000 centavos) | 10 |
| Teclado Mecánico RGB | $250.000 COP (25 000 000 centavos) | 5 |
| Mouse Inalámbrico Ergonómico | $80.000 COP (8 000 000 centavos) | 15 |

If the table already has rows, the seed is skipped and a log message is printed: `Productos ya existen en la BD. Omitiendo seed.`

## Currency convention

All monetary values are stored and transmitted in **centavos (COP)**. Column and variable names use the `_in_cents` suffix to make the unit explicit:

```
$150.000 COP = 15 000 000 centavos
$3.000 COP   =    300 000 centavos
$2.000 COP   =    200 000 centavos
```

The frontend converts centavos to display currency using `formatCurrency(cents)` from `card.utils.ts`, which uses `Intl.NumberFormat` with `es-CO` locale and `COP` currency.
